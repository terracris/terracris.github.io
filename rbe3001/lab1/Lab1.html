Introduction

In RBE 3001, we use a serial manipulator that consists of three degrees of freedom (DoF) controlled by three servo motors that rotate to control three rotational joins, as seen in Figure 1. In addition, the serial robot arm has a small gripper servo that allows the arm to grab objects. The gripper position, or end-effector position, can be modified by adjusting the three servo motor positions at the joints. In this lab, we gained an understanding of the system architecture and communication protocols of the robot arm by using an object-oriented approach in MATLAB to write commands that control the arm and monitor its position in joint space. The commands are sent to the robot, actuate the robot arm, and characterize the timing of the system and joint-level motion trajectories.

Development and Implementation

Object Methods

For the joint space control methods, our team referenced the protocols document to determine the data format of packets sent and received between the robot and the computer. For example, the servo_jp() and interpolation_jp() functions utilized the protocol document’s details on how a command can be sent with the duration of movement, mode of interpolation, and each motor’s target position, to command the robot to a certain configuration.

Figure 2: A snapshot of the protocol.md file showing the packet structure for setting motor joint positions.

For the interpolate_jp() function, an empty data packet was constructed and filled with the command ID (1848), the specified interpolation time, a zero to indicate linear interpolation, and the three motor target positions. The servo_jp() function is an exact copy, except instead of a specified interpolation time, the duration is marked as zero because no interpolation is used. Nothing is returned from the servo_jp() or interpolate_jp() because both are simply used to command the robot, not gather information.

For the measured_js() function,  two parameters are taken in GETPOS and GET VEL. These booleans are used to determine which pieces of information are being returned.  A 2x3 matrix is initialized to all 0s because, if a piece of information is not going to be returned, then the corresponding element should be a 0. The first row is meant to be filled with positions and the second row with velocities. Next, we have two matching if statements. One checks if the position is supposed to be returned (GETPOS) and the other checks if the velocities are supposed to be returned (GETVEL). Within these if statements, the corresponding packets are read to pull the required pieces of data, either positions or velocities, and are assigned to their designated elements in the output matrix.

For the setpoint_js() function, an empty 1x3 matrix is initialized to all 0s. Then the packet containing setpoints is read from the robot. From there, the setpoints of each joint are assigned to their designated element in the output matrix.

For the goal_js() function, we created a 1x3 array property within our Robot class that stores the end-of-motion (EoM) joint setpoint positions in degrees. This property is set every time the write() function is called, which writes position and timing commands to the controller. Storing the EoM positions means that we do not have to request the data from the controller reducing unnecessary operations. The function goal_js() returns this stored property.

The blue line represents the first run compared to the purple line representing the second. The discrepancy is likely due to the slight differences in calibration when the robots are reset. Additionally, the purple line is generally slightly above the light blue line at all values. This is also due to the fact that the calibration for each run is slightly different.

Figure 4: Graphs of the joint positions over time when interpolation is not used. The bottom right histogram shows the difference in timestep during the movement.

Figure 4 shows the joint positions for the robot when no interpolation is used. In the top left graph, the base joint moves to the commanded position in approximately 0.5 seconds. There is a noticeable difference between the two runs, and there is overshoot shortly before the robot reaches the goal set point.

The difference between the two runs is significantly greater than the difference between the two runs with interpolation. This is due to the lack of intermediate control over the robot’s motions without interpolation. When interpolation is used, the robot has to hit specific points in between the origin and the goal set point, so it has to follow a tighter path. Additionally, the path of the robot’s movement without interpolation shows that the robot will overcompensate and go slightly past the set point, then settle at the set point. When interpolation is used, this does not occur. Overall, interpolation provides more accurate and precise movement when moving to a set point. Interpolation should be used when a movement is required to be highly accurate across multiple runs. Movement without interpolation can be used when the arm is required to reach a set point as fast as possible, and the accuracy of where it travels to get there is less important than the end result.

Analysis of Joint Motion Trials

        The three trials shown above display the difference between using linear interpolation and no interpolation. The joints using linear interpolation makes the joints move at a steady velocity and which forms a nearly straight position line. In contrast, the joints not using interpolation form curves that are more characteristic of a PID loop.  The overshoot and subsequent correction in joints 1 and 2 (Top graphs) in each trial demonstrate this kind of behavior. It is also interesting to note that the third joint did not display the overshoot behavior the first two joints did, even without interpolation.  The joints using interpolation avoid this overshoot behavior by planning out the steps it takes to get to the setpoint.

        One disadvantage of using interpolation is that it does slow down the motion of the joints in order to obtain a constant velocity. The joints not using interpolation typically finished their movement in well under half a second, while the joints using interpolation finished their motions around the 2-second mark, as intended, though they were less likely to have fully reached their setpoint position.

        The histograms show the time in between measurements is centered very closely to 0 with a few outliers taking longer to cycle through.

        

Discussion

Our experiments highlighted the difference between a robot arm achieving a certain position with an interpolation time and one without. Interpolation cuts down on the differences between trials. Using interpolation may take more time depending on the interpolation time you pass in, but it comes with the benefit of predictable smooth movements and a general lack of overshoot. Therefore, we recommend using interpolation to achieve more controlled and accurate positioning with minimal overshoot unless the application demands a faster response. The challenge with using an interpolated value is knowing what time to give the certain operation to avoid overshoot.

One challenge we encountered during the lab was inconsistency when calibrating. Sometimes the robot would move as expected when the script was run, and other times it would not move, even after being calibrated. We believe a more robust calibration routine is needed to provide consistency when testing and to save time. Additionally, splitting our code up into distinct sections would improve our efficiency since we reset the arm every time we changed the code, even if the changes were only to the data analysis.

Conclusion

In conclusion, we suggest creating a homing method for our robot class that sets the robot to a known preconfigured position to avoid having to calibrate the robot between runs. This would cut down on the total time spent working on the labs. Additionally, for upcoming labs, we will make more use of interpolated positioning than non-interpolated positioning due to its ability to accurately position our robot with minimal overshoot. Our next steps are to determine the appropriate time for a given operation.
