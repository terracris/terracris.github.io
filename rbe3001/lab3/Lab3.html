Introduction

In our previous lab, we used Forward Kinematics (FK) to calculate the robot’s end effector position in the task space given the joint variables, and implemented several FK methods in our robot class which accepted joint angles as input. In this lab, we will now implement Inverse Kinematics (IK) where we calculate the joint angles required to achieve a given end effector position. By sending these joint angles to the robot arm, the end effector will move to the desired position. Additionally, we will implement trajectory implementation techniques in the joint space to create a smooth trajectory between two vertices. Our final goal will be to integrate IK with the trajectory generation technique to achieve smooth motion between setpoints in the task space. In this lab, we will only focus on moving the end effector to a specific x, y, z position in the task space with no attention given to the orientation since our robot arm only has three degrees of freedom (DoF) and therefore not enough DoF to achieve both position and orientation. Our solution for inverse kinematics will be based on the diagram presented in Figure 1.

        From here we can break the arm x coordinate into 2 parts where x2 is the x component of link2 and x3 is the x component of link3. X2 is equal to sin(2) * l2, and z2 is equal to cos(2) *l2. Looking at link 3, the angle formed with the horizontal can be described as  + , which makes x3 equal to cos(+) * l3 and z3 equal to -sin(+) * l3.  Combining these two components and adding in the z offset of l0 and l1 gives you the following equations when  is equal to 0 .

 x = sin(θ2) * l2 + cos(+) * l3                     z = l0 + l1 + cos(θ2) *l2 - sin(+) * l3

Since d is equal to x when  is equal to 0 then the following equations can be derived from the earlier equations.

d = sin(θ2) * l2 + cos(+) * l3    

x = sin() * (sin(θ2) * l2 + cos(+) * l3)

y = cos() * (sin(θ2) * l2 + cos(+) * l3)

z = l0 + l1 + cos(θ2) *l2 - sin(+) * l3

Section 3 (Validating IK Solution)

        Ik3001 takes in the x, y, and z coordinates of the end effector and outputs the angles of the joints. fk3001, which we verified in the last lab, does the opposite, it takes in the joint angles and outputs the x, y, and z coordinates. We took advantage of this relationship and decided to test our ik3001 function by passing in x, y, and z coordinates into ik3001 and using the outputted joint angles as inputs into the fk3001 function. The outputted x, y, and z coordinates from the fk3001 function should match the x, y, and z values passed into the ik3001 function. This process verified the ik3001 function accuracy.

Discussion

We were able to accomplish our goals for the lab, despite several challenges we had to work through. The biggest challenge we encountered was our IK function, as we discovered on the last day of lab that we wrote it incorrectly. We did not solve out the equations fully and instead used vpasolve. This slowed down all functions referencing it significantly, thus making our data for some sections highly inaccurate as the graphs only included fewer than ten data points. For future labs, we will correct the IK function in order to calculate the joint angles efficiently.

Despite this, our cubic and quintic trajectory planning, when run off of the joint space variables, provided smooth movements of the arm. The joint space trajectory planning created curved motion in the task space domain, while the task space trajectory planning provided linear motion in the task space as that is what space the trajectory was being calculated for.

I would recommend doing an early check as to whether the ik3001 function is using a solve function. The early sign offs did not indicate that we had anything to fix, but it became apparent later on that this mistake would cause problems later on.

Conclusion

In this lab, we developed the functionality to calculate the inverse kinematics of the robotic arm’s motion by converting the coordinate position in xyz coordinates to the joint angles. Using this function, we were able to develop rudimentary trajectory planning with cubic and quintic solutions. However, we were not able to draw valid conclusions from our data since our inverse kinematics function was written incorrectly, leading to a massive loss of efficiency and a lack of accurate data for task space graphs. Before future labs, we will correct this error so we can calculate the joint angles efficiently.

The slowness of the ik3001 function made our data very jagged and angular for the cubic and quintic function that were utilizing the taskspace variables for trajectory planning. Had the ik3001 function worked faster, we likely would have been able to identify the difference in smoothness between the cubic and the quintic trajectory planning.
