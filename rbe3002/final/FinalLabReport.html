Introduction

        Our primary objective for this lab was to develop software necessary for a Turtlebot to map, localize, and navigate an unknown maze autonomously. Specifically, the Turtlebot had to be able to perform the following:

  1. Localize itself in the maze
  2. Traverse the maze while avoiding obstacles
  3. Identify frontiers of exploration to navigate to
  4. Explore and map the entire maze
  5. Navigate back to any point in the map

These goals were divided into three phases. Phase 1 involved exploring and mapping the unknown maze, Phase 2 involved navigating back to any point throughout the maze, and Phase 3 involved the robot localizing itself after being “kidnapped” to any given point in the maze.

To accomplish this, we combined and adapted key functionality developed in previous labs, such as A* pathfinding, C-Space calculation, and drive control. We combined these features into a single project, and then we integrated new functionality, such as frontier detection, mapping, and localization. Each of these functionalities proved necessary for the goals of the lab.

Methodology

        The solution implemented for the final demonstration of this project makes good use of the ROS system for node-based computation. That is, we had multiple nodes running in parallel communicating through various topics. Originally, our architecture was comprised of three nodes: map_explorer, drive_controller, and path_planner. Based on feedback from our preliminary design review (PDR) and issues with computation speed, the map_explorer node was reduced in functionality. Its displaced tasks were sorted into the new map_manager node and preexisting path_planner and drive_controller nodes. Figure 1 visualizes the functionality division:

Figure 1: Node architecture used for the final project.

The map_explorer node is the high-level control node that coordinates the current objective of the robot and the necessary actions to achieve it. Using ROS services and subscriptions, it gathers

information about the map, like the frontiers and their reachability, and commands the drive_controller accordingly. See Results for detail on frontier processing. Generally, it attempts to direct the robot to the closest frontier to the one previously explored. This results in behavior where the robot can effectively fully explore each area it encounters. It approximates frontier tracking, where the frontier is treated as a moving line instead of separate entities in discrete time steps. Its control flow chart is shown in Figure 2:

Figure 2: map_explorer control flow chart

The path_planner node is a collection of methods related to planning a path through a map. Its main feature is the modified A* algorithm developed to plan paths that follow the general trajectory of an optimal path while maintaining equal distance from the closest walls wherever possible. Prioritizing caution reduced the need for precision in both mapping and localizing by preventing conditions with low margin for error. Other functionality includes methods for finding the closest unoccupied grid cell and the closest point in a list to a given position. Most calls to these are done by map_explorer.

The map_manager node centralizes computation done on the map provided by the map server: c-space, frontier identification, and distribution of the results to other nodes. It utilizes the opencv library for efficiency in these image-based calculations.

The drive_controller node manages the robot’s driving along paths passed to it by map_explorer. Originally, it only stored the next point in the path, and was fed each point individually. The intention was to ensure that the robot never traveled a path that had been generated before its navigability was determined, which could lead to collisions. However, by sending the full path, map_explorer was free to manage other computation while the path was traversed, and the robot’s travel efficiency was improved: a more complex drive controller could follow the path smoothly and did not need to wait to receive each next waypoint. Planned functionality would have allowed the drive control to factor in the distance to walls using LIDAR readings, but this was deemed to be not worth the development time.

In the course of our project, various nodes were strategically executed at different stages to ensure the successful mapping of the environment. For mapping, we ran map_explorer, map_manager, path_planner, drive_controller, and turlebot3_slam_gmapping. The turtlebot3_slam_gmapping node encapsulated the Simultaneous Localization and Mapping (SLAM) functionality, a critical aspect of our mapping strategy. It employed sensor data, such as laser scans and odometry, to concurrently localize the robot within the environment and generate a map of its surroundings. Our frontier search algorithm prioritizes navigating to frontiers closest to the previous frontier.

Figure 4: Rqt graph visualizing the ROS nodes within our localization application.

In the last phase of our project, we move towards localization. First we must store the map. After generating a map, we run the map_saver node. Once the map_saver node has been executed and the map has been successfully saved as both a YAML configuration file and a Portable Graymap (PGM) image file, the next step involves making this map available for real-time use. This is where the map_server node comes into play. The map_server node functions as a map server within the ROS framework, allowing other nodes to dynamically access and utilize the stored map during runtime. After the map is running, the robot is “kidnapped,” or placed at a random position in the map. Now we begin localization. First, we run the Adaptive Monte Carlo Localization (AMCL) which uses a beam motion model to sample particles in the map. We call the AMCL service global_localization, which first distributes particles randomly throughout the map space. Afterwards, the localization node takes control. The localization node subscribes to the topic amcl_pose, which publishes the estimated pose of the robot on the map with covariance. To localize, our robot spins in place until the covariance of the x, y, and yaw angle are within a threshold. We found that a covariance threshold of 0.10 was sufficient for localizing our robot. A tighter tolerance can improve the accuracy of the localization of our robot. The localization node publishes String messages representing the angular velocity of the robot until this threshold is met. The drive controller subscribes to messages published by the localization node and rotates. After our robot localizes, we are instructed to command the robot to a certain point in the map. We use our pose_listener node to call our path_planner service to generate the path for the drive_controller node to follow.

Results

The results of our SLAM (Simultaneous Localization and Mapping) project demonstrate the successful achievement of our primary objectives. Throughout the three phases of the lab, our Turtlebot successfully mapped, navigated, and localized within an unknown maze autonomously. The key goals, as outlined in the introduction, were effectively addressed in each phase:

Phase 1: Exploration and Mapping

The map_explorer, map_manager, path_planner, and drive_controller nodes collaborated seamlessly to explore and map the unknown maze. Using Breadth First Search (BFS) originating from the current position of the robot, we would search until we found a neighbor of the current cell that was unexplored. We made sure to exclude cells that were not navigable. Afterward, we used dilation and erosion to group frontiers and remove “holes” in frontiers, as seen in Figure 5. One of the issues we accounted for was that sometimes the frontier centroid would end up being inside an obstacle as well. Therefore, we developed a method called closest_free_space() to find the closest navigable point to the centroid of the frontier. Our frontier search algorithm prioritized navigating to frontiers closest to the previous frontier, enhancing exploration efficiency.

Figure 5: Frontier detection and grouping. (Left to right: raw, dilated, eroded)

Figure 6: Frontier locating and path generation to frontier.

During the application of dilation and erosion for frontier detection in our project, we encountered several noteworthy challenges. A notable issue involved the merging of neighboring exploration boundaries, resulting from the dilation process. This merging posed a significant obstacle to the accurate identification of distinct frontiers and adversely affected the efficacy of our path planning endeavors. Additionally, the dilation process introduced distortions in the shape and size of the frontiers, consequently impacting the precision of navigation along these boundaries. Early in our experimentation phase, we confronted the challenge of frontier merging. However, through the implementation of our algorithm designed to navigate to the closest frontier in relation to the previous one, coupled with a resampling for frontiers, we successfully mitigated and resolved this issue. To resolve potential distortions, we used a 3x3 matrix seen in Figure 7, for both dilation and erosion.

Figure 7: Kernel used for Dilation and Erosion.

Another noteworthy challenge we encountered was considering the transformations of coordinates between frames. When we commanded the robot to move, we were sending coordinates relative to the map frame and not the odom frame. This resulted in the robot drifting and creating an error in the expected odometry and the actual odometry. Therefore, we had to transform the pose coming in from the path/map frame to the odom frame to ensure the robot navigated appropriately.

Phase 2: Navigation back Home after mapping

After we mapped all frontiers, our robot navigated back to its starting position. So, we stored the starting location of the robot and generated a path using our path_planner service which our robot followed.

Phase 3: Localization

Using the AMCL ROS package, we successfully localized our robot. In our project, we had to modify the amcl_diff launch (Adaptive Monte Carlo Localization for Differential Drive) file. We changed the laser_model_type to beam instead of using the likelihood field. In addition, we decreased the total number of particles used to increased the speed of our localization. The default amcl_diff launch file had a minimum of 500 particles and maximum of 5000. We reduced the number of particles by a factor of 5. We found that a minimum of 100 particles and a maximum of 1000 particles gave us successful results with improved timing. We localized our robot within 30 seconds with this strategy. When we commanded our robot to spin, it rotated with an angular velocity of 0.6 radians per second.

Discussion

The goal of this lab was to develop robot control software for general mapping and navigation in a controlled environment. Our implementation met that goal by providing the following functionality:

Mapping, using the ROS Gmapping packages. The map is an occupancy grid generated from LIDAR readings and saved for future use.

Localization, using the ROS Gmapping and AMCL packages. Gmapping handles localization when generating the map, and AMCL when using a static map.

Frontier Identification/Analysis, through a search that only finds reachable frontiers and the use of image processing techniques to speed up processing time.

Optimal Path Planning, with a modified version of the A* algorithm that helped mitigate wall collisions. Technically, the modifications mean that paths are not always optimal by purely distance, but optimal paths would increase the challenge of other aspects of navigation.

Navigation, using a dedicated ROS node to handle robot control along a planned path while other nodes monitor the updating map. The full stack of path planning, optimizing, and driving algorithms generate smooth motion with efficient turning. Odometry and localization allow for closed-loop navigation that can correct for errors.

        Combined, this implementation met the requirements laid out in the Lab 4 handout document. Our demonstration successfully mapped the maze, navigated back to the start, and localized after being “kidnapped” within the allowed time limit.

Conclusion

        Overall, this lab provided a number of key learnings and valuable experience with robotic navigation. Implementing the navigation stack ourselves helped provide an excellent understanding of the different components that enable robotic navigation and how each component works. Integrating each of these components in ROS also provided valuable experience with ROS, which will prove very useful given its prevalence in the industry. Additionally, the success of the final product showed us how effective these methodologies can be for the purpose of navigation. The scope of the project also taught us a number of key lessons about software development for large projects, such as the importance of prior planning, maintaining an organized software architecture, minimizing duplicate code, and good source-control practices. Overall, this project provided an excellent educational experience and culminated in a successful final product.
