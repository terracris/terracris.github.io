Introduction

In this lab, we used ROS to implement key functionality with the goal of allowing our Turtlebot to efficiently navigate from any point on a given map to any other. We implemented C-Space calculation to determine the areas the robot can traverse, A* pathfinding to allow the robot to find and follow the optimal path through this space between any two points, and trajectory optimization to simplify the path produced. The effectiveness of this functionality is crucial, as it will serve as a base for more complex functionality later on in the project.

Methodology

        The code written for this lab relies much more heavily on certain ROS functionality than previous labs. Two existing services are utilized: the map_server, to provide map information to the other nodes, and the tf service, which handles spatial relationships and transformations. Both of these must be started alongside roscore for use. This lab assumed the map frame to be ground truth, so the publisher node broadcasts static values specifying an identity transformation.

        The main objective of the lab was to write a custom service to provide path planning. To interface with it, a custom listener node was written that watches for Rviz commands and requests a path based on them. This must also be running for full functionality.

        Typical environment setup and program usage would be as follows:

  • A system environment with access to the ROS installation and Python would contain the .py files and launch file in a ros package named lab3
  • Rviz and Gazebo would be installed and accessible
  • If using the simulation, the launch file would start Gazebo. If not, the physical robot would be on and connected, and Gazebo would not be launched
  • The program would be run with the command: roslaunch lab3 lab3.launch

To generate a path and command the robot to move, the Rviz 2D Nav Goal tool should be used to select the start (robot) and end position. Rviz shows the obstacles in black, the c-space in green, and navigable cells in gray. The path appears as a red line and a set of waypoint cells in yellow that show a minimal representation of the optimized path.

Discussion

Our goals in this lab were to implement path planning for a mobile robot using C-Space calculation to determine areas the robot can traverse and A* pathfinding to and optimize our path. As you can see in Figures 1-3, we successfully generated our C-Space, path, and optimized path. C-Space represents the feasible configurations of the robot in its environment. Calculating the C-Space is essential for collision avoidance, ensuring that the robot plans its path while avoiding obstacles or collisions with the environment. We used A* for our path planning, a widely used pathfinding algorithm that helps the robot find the optimal path from its starting point to the goal while considering the cost associated with different paths. This ensures that the robot reaches its destination efficiently and optimally. Generating an optimized path is also crucial for improving the efficiency of the robot’s movement. An optimized path minimizes the time and resources required for the robot to reach its destination.

In our implementation, we used a dictionary called cost_so_far, and came_from to keep track of our path. In addition, we used a min priority queue to determine what nodes we would visit based on a priority rating. To determine the cost to a certain cell in our occupancy grid, we calculated the Euclidean distance with our static method euclidian_distance(), which calculates the distance between two points, between the current node/cell we were at to the neighboring cell. The priority of a cell within our priority queue was based on the edge cost + heuristic. Our heuristic was the Euclidean distance between the neighboring node and the goal. We continued exploring the map until we reached our goal, as seen in Figure 3.

At first, when we calculated edge cost, we considered all neighboring cells to be + 1 distance away from the current cell observed. This resulted in suboptimal A* path generations. Additionally, this increased the number of cells explored using A*.  However, afterward, we realized that diagonal motions were traveling on the hypotenuse of a grid cell. Therefore, horizontal and vertical movements were considered +1 cell. However, diagonal paths were ~1.4 cells worth of travel. This improved the performance of our A*.

The planned path is returned from the A* method in the form of a list of grid coordinates to travel to. On its own, this would work for actual navigation, but it’s not the most efficient strategy. To extract waypoints and optimize the path in one step, each point is iterated through, and the slope to the previous point is calculated. For each step where the slope calculated is different from the slope before it, that point is a vertex and is added to the waypoint list. This removes every intermediate point that falls on the line between its neighbors. The start and end points are also included unconditionally.

Our launch file contained 5 nodes:

  • lab3: Runs the Python code written for this lab in path_planner.py
  • map_server: Runs the built-in map service
  • tf: Runs the built-in transformation handling service
  • rviz: Opens Rviz with the configuration file created for this lab
  • lab3(2): Runs the listener that handles the path request service

It also included two other launch files:

  • Turtlebot3 bringup launch file
  • Turtlebot3 gazebo environment

In this lab, we created client and server nodes. We utilize a client service caller named "Listener," which publishes to the '/plan_path' topic. The client subscribes to the '/move_base_simple/goal' topic, and its callback method, self.get_path_callback, is invoked. This callback function manages the call to the Service node, path_planner, initiating the plan_path() method within the service node. The client sends a service request using GetPlanRequest, a message type containing starting and goal positions. Our client node then requests our map server service, which converts our map into an occupancy grid. Once we obtain the map as an occupancy grid, we perform C-space calculations and employ our A* path planning implementation to generate the path.
